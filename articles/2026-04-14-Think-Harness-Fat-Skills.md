## 核心观点

用 AI 编程 Agent 的人比用 Cursor/Chat 的工程师效率高 10-100 倍。但关键不是模型更聪明，而是**架构**——Thin Harness, Fat Skills。

---

## Harness Engineering (缰绳工程)

Anthropic 不小心把 Claude Code 的 512,000 行源码发布到 npm 了。Garry 读了之后确认：
**秘密不在模型，而在包装模型的东西。**

实时仓库上下文、Prompt 缓存、专用工具、上下文精简、结构化会话记忆 —— 这些给模型在正确的时间提供正确的上下文，同时不被噪声淹没。

---

## 五个核心定义

### 1. Skill 文件 (技能文件)
可重用的 Markdown 文档，教模型"怎么做"，不是"做什么"。
• 像方法调用，接受参数。同一个 /investigate 技能，指向不同数据源，就变成完全不同的分析师。
• **Markdown is the programming language. Human judgment is the runtime.**

### 2. Harness (缰绳)
运行 LLM 的程序，只做四件事：循环运行模型、读写文件、管理上下文、执行安全。
**反模式**："厚缰绳薄技能" —— 40+ 工具定义吃掉一半上下文窗口。
**正确做法**：快而窄的专用工具。Playwright CLI 每次操作 100 毫秒，而不是 MCP 往返 15 秒。

### 3. Resolvers (解析器/路由器)
上下文的路由表。当任务类型 X 出现时，先加载文档 Y。
Garry 的 CLAUDE.md 曾经有 20,000 行，结果模型注意力下降。Claude Code 告诉他砍掉。修复方案：约 200 行，只是文档指针。解析器在需要时加载正确的文档。

### 4. Latent vs Deterministic (潜在空间 vs 确定性)
• **潜在空间**：智能所在 (模型读取、解释、决定)
• **确定性**：信任所在 (SQL、编译代码、算术)
**最好的系统残酷地把正确的工作放在正确的地方。**

### 5. Diarization (人物/主题档案化)
模型阅读关于某个主题的一切，写出结构化档案——从几十上百份文件中提炼出**一页纸的判断**。没有 SQL 查询能产生这个。

---

## 三层架构

• **顶层：Fat Skills** (Markdown 流程，编码判断、过程和领域知识) —— **90% 的价值在这里**
• **中层：Thin CLI Harness** (约 200 行代码。JSON 进，文本出。默认只读)
• **底层：应用程序** (QueryDB、ReadDoc、Search、Timeline)

## YC 实际系统案例：Startup School 6000 创始人匹配

传统方法：15 人团队看申请，凭直觉判断，更新电子表格。在 200 人时有效，6000 人就崩溃。

**新系统：**
• `/enrich` 技能拉取所有来源，运行档案化，捕捉创始人"说的"和"实际在做"之间的差距。
• `/match` 技能用不同策略进行分组匹配，模型做聚类算法永远做不了的判断。
• `/improve` 技能读取 NPS 调查，重写技能文件自身。

**结果**：7 月活动 12% "OK" 评分。下一次活动 4%。**技能文件学会了"OK"的真正含义，系统自动变好了，没人重写代码。**

## 永久升级原则

"如果你让我做一件事，而且这件事将来还会发生，你必须：第一次手动做 3-10 个样本。给我看输出。如果我批准，把它编码成技能文件。如果需要自动运行，放到 cron 上。测试标准：如果我必须让你做第二次，你就失败了。"

**这就是 100 倍效率的来源。不是更聪明的模型。厚技能、薄缰绳，以及编码一切的纪律。**
