# 三省六部幻觉：为什么"虚拟公司"式多Agent架构在工程上不成立

**作者**: @sujingshen (SagaSu)
**来源**: [X (Twitter)](https://x.com/sujingshen/status/2043898494818410731?s=46)
**日期**: 2026-04-14
**类型**: 文章

---

## 核心结论

把多个 Agent 分别命名为"产品经理"、"架构师"、"测试工程师"，像公司部门一样传递文档、协作完成任务——这种模式（即 CrewAI、MetaGPT 式的角色分工）在工程上有根本缺陷。**Anthropic、OpenAI、Google 三家构建生产级 Agent 系统时，没有一家采用这个模式。**

---

## 为什么"虚拟公司"模式会失败

### 1. 解决的是人类的瓶颈，不是 AI 的瓶颈

人类需要分工是因为：
- 单个人的注意力有限，无法同时处理所有信息
- 人有专业壁垒，学习切换成本高
- 人与人之间需要接口来协调

但 LLM 的特性完全不同：
- 同一个模型既能写 PRD 又能写代码，没有"职业边界"
- 模型的瓶颈不是注意力广度，而是推理深度和信息完整性
- 模型之间没有"文化"和"默契"来补偿信息损耗

### 2. 角色标签制造了假边界

给 Agent 贴上"测试工程师"标签，会让它看到架构问题时因为"不在职责内"而直接跳过，**封死了最有价值的跨边界推理**。

### 3. 信息在流转中死亡

Agent A 产出文档给 Agent B，传递的是**结论而不是推理过程**。每次传递都在累积误差，导致最终输出"局部正确但整体漂移"。人类组织靠会议和默契补偿信息损耗，Agent 之间没有这些机制。

---

## 三大厂商的真实做法

### Anthropic: Context Engineering + 显式状态文件

- 使用 `progress.txt` 作为跨 session 的工作日志
- 用 Git history 作为状态锚点
- 架构是 **Orchestrator-Worker**（主从模式）：一个主 Agent 分解任务，多个 Subagent 并行探索不同方向，结果回流给主 Agent 综合
- 本质是"同时撒网捞鱼"（并行搜索），而不是"接力赛"（分工传递）

### OpenAI: Compaction + Skills + 结构化 Spec 文件

- 使用 `runbook` 文件作为共享记忆和审计日志
- 通过 `previous_response_id` 让模型在同一线程继续工作，而不是每次重建上下文
- 引入"Skills"概念：可复用的指令集挂载进容器，而不是给 Agent 贴角色标签

### Google: 大窗口 + Context-driven Development

- 使用 Conductor 扩展，把项目意图从聊天窗口移入代码库里的持久化 Markdown 文件
- 引入"Thought Signatures"（思想签名）机制：在长 session 里保存推理链关键节点，防止逻辑前后不一致

---

## 正确的多 Agent 架构原则

1. **推理链不能断，只能分叉再合并**：多 Agent 的正确用法是主 Agent 持有完整意图，子调用是为了深挖子问题，结果回流给主 Agent，而不是传给下一个 Agent。

2. **多 Agent 的价值是并行覆盖，不是分工**：性能提升主要来自"花了更多 token 覆盖更大的搜索空间"，而不是"分工更合理"。适合广度优先探索，不适合深度连续推理。

3. **验证 Agent 是找茬者，不是接棒者**：如果做多 Agent 质量控制，应该让一个 Agent 专门找另一个 Agent 的问题（对抗性检验），而不是传递工作成果。

4. **工具决定了能力，角色只是约束**：给 Agent 配什么工具（bash、文件读写等）远比给它贴什么标签重要。

---

## 实践建议

- 不要问"我需要几个 Agent"，要问"**任务的信息依赖结构是什么**"。
- **连续推理/高度依赖上下文**的场景（如写复杂功能设计）：单 Agent + 好的 Context Engineering 优于多 Agent。
- **同时探索多个独立方向**的场景（如研究 10 个竞品）：多 Agent 并行是合理的，因为信息损耗代价最小。
- **跨 session 任务必须使用外部状态文件**，应包含四类信息：
  - **任务目标**（不变，防止漂移）
  - **已完成的步骤**（追加，保留完整历史）
  - **当前状态**（覆盖，反映最新进展）
  - **已知的坑**（追加，避免重复踩）

---

## 总结

"三省六部"模式之所以流行，是因为它好解释、好展示，满足了管理层对"AI 像团队一样工作"的想象。但它隐藏了上下文传递的巨大损耗。等到任务复杂度上来，系统开始"局部正确整体错误"时，问题才会暴露。

> 等你发现问题的时候，流水线已经很长了。
