## 核心观点

Claude Code 的上下文窗口已扩展至 **100 万 Token**。很多人认为这意味着“再也不用担心上下文不够用了”，但实际上，**上下文腐烂 (Context Rot)** 依然存在。问题的核心不再是“上下文不足”，而是“如何管理上下文”。

在 Agent 完成一次操作后，你面临 5 个选择：**Continue、Rewind、Clear、Compact、Subagent**。选对路径，才能让 AI 保持最佳状态。

---

## 1. 为什么 1M 上下文依然会“变笨”？

*   **上下文腐烂 (Context Rot)**：上下文越大，模型的注意力越容易分散。旧的不相关信息会干扰当前的任务。
*   **失败的痕迹**：如果你总是说“不行，换个方法”，这些失败的尝试会留在上下文里，占用 Token 并分散注意力。
*   **自动压缩 (Autocompact) 的陷阱**：模型自动总结时，可能会在不恰当的时机丢弃关键信息（比如只保留了调试信息，丢了业务逻辑）。

**结论**：1M 上下文不是让你无脑塞满内容的，而是给你留出**主动压缩 (Proactive Compaction)** 的缓冲空间。

---

## 2. 核心工具：/usage 面板

新版 Claude Code 推出了 `/usage` 命令，可以查看剩余上下文和使用量。
**建议**：养成在感觉对话变长时敲一下 `/usage` 的习惯。
*   如果还有大量剩余 -> 考虑主动使用 `/compact <hint>`。
*   如果已经快满了 -> 考虑 `/clear` 开启新会话。

## 3. 会话管理的 5 种选择

每当 Claude 完成一个操作，你需要根据情况从以下 5 个选项中做决定：

### ① Continue（继续）
*   **适用场景**：当前的上下文信息依然全部是“承重”的（load-bearing），没有废话。
*   **判断标准**：如果不保留这些信息，重新解释的成本是否大于继续对话的成本？如果是，就继续。

### ② Rewind（回滚）
*   **命令**：`Esc` 双击 或 `/rewind`。
*   **适用场景**：Agent 做错了，或者方向偏了。
*   **核心逻辑**：与其说“刚才那个不行，试 X"，不如直接回滚到做决定之前的那个节点。**把错误的痕迹从上下文中切除**，只保留正确的意图。
*   **Onevcat 点评**：以前以为 Rewind 只是简单的撤销，实际上它是**清理上下文垃圾**的最佳手段。

### ③ Clear（清除/新会话）
*   **命令**：`/clear`。
*   **适用场景**：任务已经完全改变，或者上下文已经太乱，修补不如重来。
*   **最佳实践**：不要直接清空。先让 AI 写一个简短的 **Brief（简报）**，总结当前的进度、已知的坑、下一步计划。然后把这个 Brief 贴到新会话的开头。

### ④ Compact（压缩）
*   **命令**：`/compact` 或 `/compact <hint>`。
*   **适用场景**：上下文快满了，但任务还没结束。
*   **核心逻辑**：强制模型对当前状态进行总结。**一定要带 hint（提示）**！
    *   错误用法：`/compact`（模型可能把重要的测试细节丢了）。
    *   正确用法：`/compact 保留当前的数据库结构和测试失败的堆栈信息，其他可以总结`。
*   **防范 Bad Autocompact**：在任务间隙主动压缩，避免模型在你专心调试时突然自动压缩导致信息丢失。

### ⑤ Subagent（子代理）
*   **适用场景**：有一个相对独立、不需要主会话所有上下文的小任务（比如写个正则、查个 API 文档）。
*   **核心逻辑**：不要让主 Agent 带着 50 万 Token 的包袱去干一件只需要 1000 Token 的事。派个子代理去干，干完把结果拿回来。

---

## 4. 总结：上下文管理的心智模型

1.  **上下文不是垃圾桶**：不要在里面堆积失败的尝试。
2.  **主动优于被动**：不要等自动压缩（Autocompact）来救你，它往往会在最糟糕的时候触发。在关键节点主动 `/compact` 或 `/clear`。
3.  **Rewind 是修剪剪刀**：用 Rewind 剪掉错误的分支，保持上下文的“主干”清晰。
4.  **Brief 是过桥费**：在开启新会话（Clear）前，花一点时间生成一个 Brief，这是跨会话保持智能连续性的关键。
