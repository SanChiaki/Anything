# 我翻遍了所有的 AI 记忆工具，发现它们分为两大阵营

**作者**: witcheer ☯︎ (@witcheer)
**来源**: [X (Twitter)](https://x.com/witcheer/status/2044456778843238689?s=46)
**日期**: 2026-04-16
**类型**: 文章

---

## 核心结论

在 GitHub 上有 450+ 个标记为 "agent-memory" 的仓库。作者通过调研发现，这些工具并非同构，而是分为两个截然不同的阵营：
*   **阵营 1：记忆后端 (Memory Backends)** —— 解决“AI 应该记住什么？”的问题，侧重于事实召回。
*   **阵营 2：上下文基质 (Context Substrates)** —— 解决“AI 应该在什么上下文中工作？”的问题，侧重于上下文的积累和复合。

大多数工具（和 GitHub Stars）集中在**阵营 1**，但**阵营 2**才是真正能够支持连续、多会话、多项目工作的架构。

---

## 阵营 1：记忆后端 (Memory Backends)

这些工具从对话中提取事实，存储到向量数据库中，需要时再检索出来。
*   **核心循环**：对话 → 提取事实/存储内容 → 存入数据库 → 检索注入。
*   **智能点**：在于提取和检索算法。
*   **局限性**：解决的是“事实回忆”问题（比如“我之前说了什么”），但无法解决复杂的上下文积累。

### 代表工具：
*   **Mem0 (53.1k stars)**：采用者最多的领导者。操作简单（增删改查），但记忆是扁平的，没有关联，且不会随时间进化。
*   **MemPalace (46.2k stars)**：本地优先的逐字记忆。将对话按实体/主题分类，检索召回率极高（96.6%），但随对话量线性增长，无法压缩。
*   **Supermemory (21.8k stars)**：主打“记忆不是 RAG"。具有时间感知能力，会自动遗忘过期的旧事实（例如搬家后自动更新城市）。
*   **Honcho (2.4k stars)**：将人类和 Agent 视为“对等体”，构建心理模型，不仅仅是记忆内容，而是理解用户的思考方式。

---

## 阵营 2：上下文基质 (Context Substrates)

这些工具维护结构化、人类可读的上下文，这些上下文会跨会话积累。上下文即文件。Agent 读取文件，在其中工作，写回文件，整个过程随时间复合。
*   **核心循环**：Agent 读取结构化上下文 → 在其中工作 → 写回上下文 → 下一次会话上下文更丰富。
*   **智能点**：在于积累（Accumulation）。
*   **特点**：人类可以直接阅读、编辑、修正上下文文件。

### 代表工具：
*   **OpenClaw (358k stars)**：使用 Markdown 文件（MEMORY.md, 每日笔记, DREAMS.md）。核心理念：“模型只‘记住’保存到磁盘的东西，没有隐藏状态。”
    *   **梦境机制 (Dreaming)**：后台进程将每日笔记合并为长期记忆（轻睡筛选 -> REM 加权提升 -> 深睡合并）。
*   **Zep (4.4k stars)**：**重品牌为“上下文工程” (Context Engineering)**。这是一个强烈的市场信号，表明该领域正在转向。底层使用时序知识图谱。
*   **Thoth (145 stars)**：小型项目，架构极深。构建个人知识图谱，拥有夜间四阶段“梦境循环”进行去重、关系推断和置信度衰减。
*   **TrustGraph (2.0k stars)**：引入“上下文核心 (Context Cores)"——可移植、版本化的上下文包。将上下文视为代码，可以版本控制、测试、回滚。
*   **MemSearch (by Zilliz)**：以 Markdown 文件为主，向量搜索仅作为文件的“影子索引”。即使是最头部的向量数据库公司也承认文件才是真相来源。

---

## 趋势预测

*   **"Context Engineering" 将取代 "Memory"**：作者预测在未来 6 个月内，“上下文工程”将取代“记忆”成为描述高级 Agent 基础设施的默认术语。
*   **基质架构将胜出**：构建上下文基质的项目将超越那些仍将问题框架化为“事实存储”的项目。
*   **长期 Agent 的需求**：如果你运行的是 24/7 不间断的 Agent（在睡觉时工作，跨项目共享知识库），记忆后端不够用，必须使用上下文基质。
